# Eval 가이드

거짓된 자신감 대신 실제로 스킬을 개선하는 eval 기준을 작성하는 법.

---

## 황금 규칙

모든 eval은 yes/no 질문이어야 한다. 척도가 아니다. 느낌 체크가 아니다. Binary.

이유: 척도는 변동성을 증폭시킨다. 1-7점 척도의 eval이 4개면 총점의 분산이 엄청나게 커진다. Binary eval은 신뢰할 수 있는 신호를 준다.

---

## 좋은 eval vs 나쁜 eval

### 텍스트/카피 스킬 (뉴스레터, 트윗, 이메일, 랜딩페이지)

**나쁜 eval:**
- "글이 좋은가?" (너무 모호 — "좋다"가 뭔데?)
- "참여도 잠재력을 1-10으로 평가하라" (척도 = 신뢰 불가)
- "사람이 쓴 것처럼 들리는가?" (주관적, 일관성 없는 채점)

**좋은 eval:**
- "출력에 금지 목록의 표현이 0개인가: [game-changer, 핵심은, 가장 좋은 점은, 레벨업]?" (binary, 구체적)
- "첫 문장이 특정 시간, 장소, 또는 감각적 디테일을 참조하는가?" (binary, 확인 가능)
- "출력이 150-400단어 사이인가?" (binary, 측정 가능)
- "독자에게 정확히 무엇을 할지 알려주는 구체적 CTA로 끝나는가?" (binary, 구조적)

### 시각/디자인 스킬 (다이어그램, 이미지, 슬라이드)

**나쁜 eval:**
- "전문적으로 보이는가?" (주관적)
- "시각적 품질을 1-5로 평가하라" (척도)
- "레이아웃이 좋은가?" (모호)

**좋은 eval:**
- "이미지의 모든 텍스트가 가독성 있고 잘리거나 겹치는 단어가 없는가?" (binary, 구체적)
- "색상 팔레트가 소프트/파스텔 톤만 사용하고 네온, 밝은 빨강, 고채도 색상이 없는가?" (binary, 확인 가능)
- "레이아웃이 선형인가 — 좌→우 또는 상→하로 흐르고 흩어진 요소가 없는가?" (binary, 구조적)
- "이미지에 번호 매기기, 순번, 순차적 넘버링이 없는가?" (binary, 구체적)

### 코드/기술 스킬 (코드 생성, 설정, 스크립트)

**나쁜 eval:**
- "코드가 깨끗한가?" (주관적)
- "베스트 프랙티스를 따르는가?" (모호, 어떤 베스트 프랙티스?)

**좋은 eval:**
- "코드가 에러 없이 실행되는가?" (binary, 테스트 가능 — 실제로 실행하라)
- "출력에 TODO 또는 placeholder 코멘트가 0개인가?" (binary, grep 가능)
- "모든 함수와 변수 이름이 서술적인가 (루프 카운터 외 단일 문자 이름 없음)?" (binary, 확인 가능)
- "모든 외부 호출(API, 파일 I/O, 네트워크)에 에러 핸들링이 포함되어 있는가?" (binary, 구조적)

### 문서 스킬 (제안서, 보고서, 덱)

**나쁜 eval:**
- "포괄적인가?" (무엇에 비해?)
- "클라이언트 니즈를 다루는가?" (너무 개방적)

**좋은 eval:**
- "문서에 필수 섹션이 모두 포함되어 있는가: [목록]?" (binary, 구조적)
- "모든 주장이 구체적 숫자, 날짜, 또는 출처로 뒷받침되는가?" (binary, 확인 가능)
- "문서가 [X]페이지/단어 이하인가?" (binary, 측정 가능)
- "요약이 3문장 이하의 한 문단에 들어가는가?" (binary, 셀 수 있음)

## 흔한 실수

### 1. eval이 너무 많음
6개를 넘으면 스킬이 eval을 게임하기 시작한다 — 좋은 출력을 만드는 대신 테스트 통과에 최적화한다. 내용을 이해하지 않고 답을 외우는 학생처럼.

**해결:** 가장 중요한 3-6개를 골라라. 그것들을 모두 통과하면 출력은 아마 좋을 것이다.

### 2. 너무 좁고 경직됨
"정확히 3개의 불릿 포인트를 포함해야 함" 또는 "'왜냐하면'을 최소 2번 사용해야 함" — 이런 것들은 기술적으로 통과하지만 이상하고 부자연스러운 출력을 만든다.

**해결:** eval은 실제로 중요한 품질을 체크해야지, 임의적 구조 제약을 체크해서는 안 된다.

### 3. 겹치는 eval
eval 1이 "텍스트가 문법적으로 정확한가?"이고 eval 4가 "철자 오류가 있는가?"면 — 이것들은 겹친다. 문법 실패는 종종 철자를 포함한다. 이중 계산.

**해결:** 각 eval은 서로 다른 것을 테스트해야 한다.

### 4. 에이전트가 측정할 수 없음
"사람이 이것을 흥미롭게 느낄까?" — 에이전트는 이것을 신뢰성 있게 답할 수 없다. 거의 매번 "yes"라고 할 것이다.

**해결:** 주관적 품질을 관찰 가능한 신호로 번역하라. "흥미로움"은 이것을 의미할 수 있다: "첫 문장이 구체적 주장, 이야기, 또는 질문을 포함하는가 (일반적 진술이 아닌)?"

## eval 작성법: 3-질문 테스트

eval을 확정하기 전에 물어라:

1. **두 명의 다른 에이전트가 같은 출력을 채점하면 동의할까?** 아니면 eval이 너무 주관적이다. 재작성.
2. **스킬이 실제로 개선되지 않고도 이 eval을 게임할 수 있나?** 그러면 eval이 너무 좁다. 넓혀라.
3. **이 eval이 사용자가 실제로 신경 쓰는 것을 테스트하는가?** 아니면 버려라. 중요하지 않은 eval은 중요한 eval의 신호를 희석시킨다.

## 좋은 autoloop 실행 7체크 (v6.4 — autoresearch 흡수)

루프 종료 시 자체 채점:

1. **baseline부터 시작** — 변경 전 측정 먼저 (step 3 강제)
2. **binary eval만** — 척도·1-10 점수·"느낌" 금지
3. **한 변이씩** — 무엇이 점수를 올렸는지 분리 가능 (절대 규칙 #3·#6)
4. **완전한 로그** — keep·discard 모두 changelog에 기록 (1줄도 누락 금지)
5. **측정 가능한 개선** — baseline → final 점수 차이 명시
6. **오버핏 미발생** — 테스트 입력 외 신규 입력 1개로 검증해서 점수 유사한지 확인
7. **자율 실행** — 실험마다 컨펌 묻지 않음 (절대 멈추지 마라 룰)

eval 통과 but 실사용 품질 안 올랐다면 → eval이 나쁨. step 2 회귀.

## v2: 내용적 품질 eval

v1의 eval은 형식(포맷, 길이, 구조)에 집중한다. v2는 내용적 품질(분석 깊이, 논리 완결성, 다관점)도 binary eval로 측정할 수 있다.

핵심: "깊이 있는가?"는 나쁜 eval이다. 이것을 관찰 가능한 신호로 번역해야 한다.

### 내용적 eval 예시

**분석/리서치 스킬:**
- "출력이 2개 이상의 독립적 관점에서 주제를 다루는가?" (binary — 관점 수를 세면 됨)
- "모든 주장에 구체적 근거(숫자, 출처, 사례)가 1개 이상 동반되는가?" (binary, 확인 가능)
- "결론과 반대되는 증거나 반론이 최소 1건 언급되는가?" (binary — 있거나 없거나)
- "원인 분석이 표면 현상을 넘어 최소 1단계 깊은 원인까지 도달하는가?" (binary — "매출이 떨어졌다" vs "매출이 떨어진 이유는 X 때문이다")

**기획/전략 스킬:**
- "제안이 최소 1개의 제약 조건(비용, 시간, 인력 등)을 명시적으로 고려하는가?" (binary)
- "실패 리스크나 부작용이 최소 1건 언급되는가?" (binary)
- "실행 단계가 의존관계 순서로 배열되어 있는가 (선행 조건이 후행 단계보다 앞에)?" (binary, 순서 확인)

**주의:** 내용적 eval은 형식적 eval보다 채점 변동성이 높다. 내용적 eval을 넣을 때는 형식적 eval과 섞어서 3:3 이하 비율을 유지하라. 내용적 eval만 6개면 노이즈가 너무 커진다.

## v3: 모드별 eval 설계

autoloop는 full/light 모드가 있고, 내용적 eval의 비율이 모드 선택에 영향을 준다.

**eval 유형 분류:**

| 유형 | 판정 방식 | 예시 | 채점 (light) | 채점 (full) |
|------|----------|------|-------------|-------------|
| 형식적 | 기계적으로 확인 가능 | 길이, 포맷, 섹션 존재, 철자 | 1회 | 1회 |
| 내용적 | 판단이 필요 | 분석 깊이, 논리 완결, 다관점 | 1회 | 3회→2/3 합의 |

**모드 자동 판정:** 내용적 eval이 전체의 50%↑이면 full 모드를 권고한다. 이유: 내용적 eval은 채점 노이즈가 크기 때문에, 1회 채점으로는 keep/discard 판정이 불안정하다.

**eval 비율 가이드:**
- 형식적 only (코드, 다이어그램): light 모드. eval 3-5개.
- 형식+내용 혼합 (보고서, 기획): full 모드. 형식 2-3 + 내용 2-3 = 4-6개.
- 내용적 only (분석, 리서치): full 모드 필수. eval 3-4개. 내용적 eval만 6개면 노이즈가 과다.

**3회 채점 불일치 처리:**
- 3:0 (전원 다른 판정) → 해당 eval이 모호하다. pass/fail 조건을 더 구체적으로 재작성.
- 2:1 합의 → 정상. 다수결 적용.
- 매 실험마다 같은 eval이 2:1로 갈리면 → 그 eval 자체를 개선 대상으로 플래그.

## v4: auto_scorable eval (코드 자동 채점)

v4.0에서 도입. 형식적 eval 중 grep/regex/wc 등 코드로 판정 가능한 것을 자동 채점한다. LLM 채점 호출이 0이 되므로 속도가 비약적으로 향상된다.

### auto_scorable 판정 기준

| auto_scorable = yes | auto_scorable = no |
|---|---|
| 단어 수 범위 (`wc -w`) | 분석 깊이, 논리 완결성 |
| 특정 문자열 포함/미포함 (`grep`) | 주장의 근거 충분성 |
| 섹션 헤딩 존재 여부 (`grep "^## "`) | 다관점 반영 여부 |
| 금지 표현 0개 (`grep -c`) | 인과관계 정확성 |
| TODO/placeholder 0개 (`grep -c`) | 실행 가능성 판단 |
| 줄 수 범위 (`wc -l`) | 어조/톤 적절성 |
| 코드 실행 에러 없음 (`python ... ; echo $?`) | 코드 품질 (가독성 등) |

**핵심 원칙:** "확실히 코드로 판정할 수 있는 것만 yes." 애매하면 no. auto_scorable을 잘못 설정하면 채점 결과 자체가 무의미해진다.

### auto_script 작성법

eval당 1줄 bash/python 스크립트. 출력 파일 경로는 `$OUTPUT`으로 참조. 성공(pass) = exit 0, 실패(fail) = exit 1.

```bash
# 단어 수 150-400
wc -w < "$OUTPUT" | awk '{exit !($1 >= 150 && $1 <= 400)}'

# 금지 표현 0개
grep -cE '(game-changer|핵심은|가장 좋은 점은|레벨업)' "$OUTPUT" | awk '{exit !($1 == 0)}'

# 필수 섹션 존재
grep -q "^## 결론" "$OUTPUT"

# TODO/placeholder 0개
grep -cE '(TODO|FIXME|\[placeholder\])' "$OUTPUT" | awk '{exit !($1 == 0)}'

# Python 실행 에러 없음
python "$OUTPUT" 2>/dev/null; test $? -eq 0
```

### turbo 모드와의 관계

auto_scorable eval이 100%이면 turbo 모드 사용 가능. turbo에서는 모든 채점이 코드로 처리되므로 LLM 채점 호출이 0. 실험당 LLM 호출이 ~2회(실행 + 변이)까지 줄어든다.

## 템플릿

각 eval에 이것을 복사하라:

```
EVAL [N]: [짧은 이름]
질문: [Yes/no 질문]
Pass: ["yes"가 어떤 모습인지 — 한 문장, 구체적]
Fail: ["no"를 유발하는 것 — 한 문장, 구체적]
유형: [형식적/내용적]
auto_scorable: [yes/no]
auto_script: [코드 1줄 — auto_scorable=yes일 때만]
```

예시:

```
EVAL 1: 텍스트 가독성
질문: 출력의 모든 텍스트가 완전히 가독성 있고 잘리거나 겹치거나 끊긴 단어가 없는가?
Pass: 모든 단어가 완전하고 찡그리거나 추측하지 않아도 읽을 수 있음
Fail: 어떤 단어라도 부분적으로 숨겨지거나, 다른 요소와 겹치거나, 가장자리에서 잘림
유형: 형식적
auto_scorable: no
```

```
EVAL 2: 단어 수 제한
질문: 출력이 150-400단어 사이인가?
Pass: 단어 수가 150 이상 400 이하
Fail: 149 이하 또는 401 이상
유형: 형식적
auto_scorable: yes
auto_script: wc -w < "$OUTPUT" | awk '{exit !($1 >= 150 && $1 <= 400)}'
```

```
EVAL 3: 다관점 분석
질문: 출력이 2개 이상의 독립적 관점에서 주제를 다루는가?
Pass: 최소 2개의 서로 다른 시각/입장이 명시적으로 구분되어 있음
Fail: 단일 관점만 존재하거나 관점 구분이 불명확
유형: 내용적
auto_scorable: no
```
