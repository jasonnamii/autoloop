# 루프 메커니즘 — 4축 레퍼런스

autoloop의 실험 루프를 구동하는 8개 메커니즘과 적용 지점.

trigger-principles.md가 "변이 **내용**을 무엇으로 채울지" 결정한다면, 이 문서는 "루프를 **어떻게** 돌릴지" 결정한다. 두 문서는 상호 보완이지 대체가 아니다.

**v5.0 변경:** α 연속체 모드(모드별 if-else → α 분기 함수), 1후보 집중형 변이+대안키워드, 병렬 실행, 귀인 검증, 단계별 프로파일.

---

## 4축 구조

| 축 | 역할 | 포함 메커니즘 |
|----|------|-------------|
| **탐색** | 해 공간을 넓게 살핀다 | Tree of Thoughts, ReAct |
| **검증** | 결과를 다각도로 확인한다 | Self-Consistency, Debate |
| **구조화** | 문제를 정밀하게 분해한다 | Decomposition, Constraint Injection |
| **압축** | 불필요를 제거하고 개선한다 | Reflection Loop, RAR |

---

## α 분기표 (v5.0)

모든 메커니즘의 동작이 α(auto_scorable eval 수 / 전체 eval 수) 하나로 결정된다.

| 메커니즘 | α = 1.0 (turbo) | α ≥ 0.5 (light) | α < 0.5 (full) |
|---------|-----------------|-----------------|----------------|
| Tree of Thoughts | 후보 1개, 약점 없음 | 후보 1개 + 약점 1문장 | 후보 1개 + 약점 1문장 |
| 대안키워드 | 2개 (changelog) | 2개 (changelog) | 2개 (changelog) |
| ReAct | auto_scorable 결과 자동 | 수동 분석 | grep/수치화 분석 |
| Self-Consistency | 코드 채점 (다수결 없음) | 전 eval 1회 채점 | 내용적 eval 3회 채점 |
| Debate | 생략 | 생략 | 대안키워드 반론으로 대체 |
| Decomposition | 매트릭스 자동 | 매트릭스 생성만 | 매트릭스 + 패턴 분석 |
| Constraint Injection | trigger-principles.md | trigger-principles.md | + 효과 추적 |
| Reflection Loop | 판정 이유 1문장만 | pre 1문장 + 판정 이유 | pre 상세 + 판정 이유 |
| RAR | changelog 요약 + 귀인 검증 | changelog 요약 + 귀인 검증 | 매트릭스 + 패턴 + scorer 노이즈 + 귀인 검증 |

**새 모드 추가법:** α 범위만 정의하면 위 표에서 자동으로 동작이 결정된다.

## 메커니즘별 적용 지점

### 1. Tree of Thoughts (분기 탐색)

**원리:** 한 경로가 아니라 여러 사고 가지를 생성 → 평가 → 가지치기. 로컬 최적 해를 탈출한다.

**autoloop 적용 — step 4b (v5.0: 1후보 집중형):**

v4.0에서는 full 모드가 단일 프롬프트에 "후보 3개+약점+반론+선택"을 요구했다. v5.0에서는 모든 α에서 후보 1개만 생성하고, 대안키워드 2개를 changelog에 저장한다. 탐색 폭은 RAR이 대안키워드를 읽어 후속 실험의 후보로 재활용함으로써 보존된다.

```
실패: [E{N}] {eval 이름} — {실패 증상 1문장}
이전: {판정 이유 1문장}

이 실패를 해결하는 변이 1개를 제안하라.
- 변경: [어디를 어떻게]
- 약점: [이 변이가 실패할 수 있는 이유 1개]  ← α=1.0이면 생략
- 대안 키워드 2개 (미실행 후보 복원용)       ← changelog에 저장
```

**비용:** v4.0 full 대비 입력 토큰 ~60% 감소. 호출 수는 v4.0과 동일(1회). 탐색 폭 손실 없음(대안키워드→RAR 경로).

---

### 2. ReAct (행동 루프)

**원리:** 생각 → 행동(도구 호출) → 결과 → 다시 생각. 정적 추론을 동적 문제 해결로 전환한다.

**autoloop 적용 — step 4a 강화:**
- 실패한 출력을 grep/검색해서 패턴을 수치화한다
- auto_scorable eval의 실패는 코드 출력에서 자동으로 수치화됨
- **v5.0 추가:** step_profile에서 토큰·시간 병목을 확인하여, "어느 단계를 최적화해야 가장 큰 효과인가"를 수치로 판단

| α | ReAct 깊이 |
|---|-----------|
| = 1.0 | auto_scorable 결과 자동 |
| ≥ 0.5 | + 수동 분석 |
| < 0.5 | + grep 수치화 + 프로파일 병목 분석 |

---

### 3. Self-Consistency (다수결 채점)

**원리:** 동일 문제를 여러 경로로 풀이 → 결과 다수결 선택. 단일 추론 오류를 제거한다.

**autoloop 적용 — step 4c:**

| eval 유형 | auto_scorable | 채점 방식 | 다수결 |
|-----------|--------------|----------|--------|
| 형식적 | yes | 코드 자동 채점 | 불필요 (결정적) |
| 형식적 | no | 실행+채점 합침 (1회) | 불필요 |
| 내용적 | no | 실행 Agent 1회 + 별도 Agent 2회 | 2/3 합의 (α < 0.5만) |

**병렬 실행 (v6.2 — MUST):** N회 실행을 **동시 Agent spawn 필수**. auto_scorable은 실행 완료 즉시 코드 채점. 내용적은 전체 실행 완료 후 LLM 채점. wall-clock이 ~1/N로 줄어든다. **병렬 가능 환경에서 순차 실행 = FAIL.** Agent tool이 병렬을 지원하지 않는 환경에서만 순차 폴백 허용(사유를 changelog에 명시).

---

### 4. Debate / Adversarial (내부 토론)

**원리:** 반대 입장 생성 → 논쟁 → 통합. 편향을 제거하고 논리를 강화한다.

**autoloop 적용 — v5.0에서 대안키워드로 대체:**

v4.0에서는 단일 프롬프트 내 "각 후보 실패 반론"으로 구현했다. v5.0에서는 후보가 1개이므로 반론 단계가 제거되지만, 대안키워드 2개가 "이 후보가 실패하면 다음에 시도할 방향"을 암시적으로 제공한다. 대안키워드는 changelog→RAR 경로로 후속 실험에 반론적 역할을 수행한다.

---

### 5. Decomposition (문제 분해)

**원리:** 큰 문제 → 독립 서브문제로 분리 → 병렬 해결 → 통합. 복잡도를 감소시킨다.

**autoloop 적용 — step 4a:**
- eval × test-input 매트릭스로 실패 분해 (변경 없음)
- auto_scorable eval의 매트릭스는 코드 채점 결과에서 자동 생성

| α | 분석 깊이 |
|---|----------|
| = 1.0 | 매트릭스 자동 생성+분석 (코드) |
| ≥ 0.5 | 매트릭스 생성 유지, 분석 깊이 축소 |
| < 0.5 | 매트릭스 + 패턴 분석 (LLM) |

---

### 6. Constraint Injection (제약 주입)

**원리:** 의도적으로 규칙 삽입. 산출물의 구조적 완성도를 올린다.

**autoloop 적용 — 기존 trigger-principles.md 그대로 계승.** 주입 후 다음 실험에서 해당 eval이 개선됐는지 명시적으로 추적. 개선 없으면 주입 문장 제거.

---

### 7. Reflection Loop (자기 비평)

**원리:** 1차 결과 생성 → 스스로 오류/약점 지적 → 수정.

**Pre-reflection (step 4b 내):**
- 변이 후보의 "약점" 1문장이 pre-reflection 역할
- α = 1.0이면 생략 (코드 채점이 대체)

**Post-reflection (v5.0 — 프로파일 흡수):**
- changelog의 "판정 이유" 1문장 + **프로파일 데이터** → 다음 4a 입력
- 별도 호출 0. 정보 손실 없이 프로파일 데이터가 추가되어 오히려 정보 증가

---

### 8. RAR + 귀인 검증 (v5.0 강화)

**원리:** 기존 정보의 재배열/참조 구조화 + 대조 실험을 통한 귀인 검증.

**발동 조건:** 3회 연속 discard 또는 점수 정체.

**v5.0 — 2단계 진단:**

```
① 귀인 검증 (v5.0 신규):
   1. 현재 지목된 스킬 구간을 원본(baseline)으로 되돌린다
   2. 다른 구간의 변이는 유지한 상태에서 재실행
   3. 점수 변화 없으면 → 귀인 틀림
      → eval×input 매트릭스 재분석
      → 새 실패 패턴 식별 → 새 귀인으로 변이 방향 전환
   4. 점수 변화 있으면 → 귀인 맞음 → ②로

② 기존 RAR:
   → changelog 재분석 → 시도/미시도 매트릭스 생성:
     시도한 변이 유형     | 형식적  | 내용적
     추가 (지시 삽입)     | N회    | N회
     수정 (기존 재작성)   | N회    | N회
     제거 (불필요 삭제)   | N회    | N회
     예시 (in/out 추가)   | N회    | N회

   → "0회" 셀 + 대안키워드 미시도 항목 → 다음 변이 후보 우선
   → keep 패턴 공통점 추출 → 다른 eval에 적용
```

**비용:** 귀인 검증의 대조 실험 1회는 정상 경로에서 발동하지 않으므로 오버헤드 0. 뺑뺑이 감지 시에만 +1회 실행. 뺑뺑이 3회를 도는 것보다 대조 1회가 효율적.

**Scorer 노이즈 매트릭스:** α < 1.0일 때만 내용적 eval 추적.
- 3실험 연속 scorer 일치도 < 80% → 루프 정지 + eval 재설계 강제
- auto_scorable eval은 노이즈 0이므로 매트릭스에서 제외

---

## 메커니즘 조합 순서 (실험 루프 내, v5.0)

```
① Decomposition: eval×input 매트릭스로 실패 분해 + 이전 판정 이유 흡수 + 프로파일 병목  [step 4a]
② ReAct: 실패 출력 grep/수치화 (auto_scorable은 자동) + 프로파일 분석                   [step 4a]
③ Constraint Injection: trigger-principles.md에서 원리 탐색                               [step 4b 준비]
④ Tree of Thoughts + Reflection(pre): 1후보 + 약점 + 대안키워드                           [step 4b]
⑤ 실행 + Self-Consistency: 병렬/순차 실행 + 채점 (auto_scorable→코드, 내용적→LLM)       [step 4c]
⑥ keep/discard + 판정 이유 + 프로파일 기록                                                [step 4d]
⑦ RAR + 귀인 검증: (정체 시만) 대조실험 → changelog 재분석 → 대안키워드 소화             [step 4e]
```

v4.0 대비: ④가 토큰 ~60% 감소, ⑤가 병렬 옵션 추가, ⑦이 귀인 검증 추가.
